The 3 Critical Problems (Rejection-Level)
1. Dataset authenticity — Your synthetic sequences are PCA-projected frequency vectors from EMBER (PE headers) and MalDroid (Android syscalls), not actual Windows pre-encryption API traces. A reviewer at IEEE TIFS will flag this immediately. You need Cuckoo Sandbox dynamic analysis reports for real Windows ransomware families.
2. Only ~1,000 samples — The MHSA-LSTM was validated on 39,378 ransomware samples and 9,732 benign applications MDPI. Your 1,000 synthetic samples won't survive comparison.
3. No classical baselines — You compare QLSTM vs your old CNN-VQC. A journal needs you to compare against classical LSTM, CNN, Random Forest, and XGBoost. Without this, you cannot claim quantum advantage.

The 3 Truly Fatal Issues (Desk Rejection Risk)
#1 — Reproducibility mandate. IEEE TIFS explicitly requires authors to publish code and data used to produce figures and tables, and has drafted specific instructions for deep learning papers about enabling research reproducibility. MDPI Your data_raw.zip doesn't exist on GitHub. A reviewer who can't run your pipeline can't accept the paper. Fix: upload your synthetic dataset to Zenodo (free, gives you a citable DOI), pin all package versions in requirements.txt, and add a Docker/Conda environment file.
#2 — The fundamental dataset mismatch. EMBER is a static PE byte histogram dataset and MalDroid is Android syscalls. Neither contains Windows API call sequences. Your entire USP is detecting Windows pre-encryption API patterns. A single expert reviewer ends the paper on this point alone.
#3 — The 30–70 second claim is circular. You inject malicious behavior at t_start = randint(10, 60), then your SHAP plot shows a spike at timesteps 10–60. Of course it does — the data was constructed that way. You need to validate SHAP peaks against ground-truth injection timestamps on a held-out set where timestamps were never exposed to the model.

The Most Overlooked But Damaging Issues
#4 — No statistical significance testing. Reporting "94% F1 vs 89% F1" without a McNemar's test means nothing. The McNemar's test is the only test with acceptable Type I error for comparing classifiers when learning algorithms can only be evaluated once on a single test set. arXiv You need this for every baseline comparison in your table.
#5 — No quantum advantage proof. A large-scale benchmark study found that out-of-the-box classical models outperform quantum classifiers overall, and that removing entanglement often results in as good or better performance, suggesting quantumness may not be the crucial ingredient. Springer Reviewers in 2026 know this. You need an ablation that replaces VQCs with matched classical layers to show the quantum component is actually contributing.
#6 — No NISQ noise analysis. Your simulation is noiseless. The training of VQC parameters faces significant scalability challenges and is susceptible to hardware limitations and sampling noise inherent in the current generation of quantum devices. Emergent Mind One experiment with PennyLane's DepolarizingChannel turns this from a weakness into a strength.
#10 — The inference latency contradiction. If QLSTM takes 45 seconds to classify one sample (possible with 480 VQC evaluations), your 30–70 second lead-time is consumed by the model itself. You must benchmark and report inference latency.
#17 — Adversarial robustness. Modifying code structure or using obfuscation techniques can create adversarial samples that alter surface features, making malware similar to legitimate software and confusing machine learning detection models. MDPI Real ransomware operators already do this. A 2026 paper ignoring adversarial evasion looks dated.
#18 — MITRE ATT&CK alignment. One table mapping your detectable API calls to specific ATT&CK techniques (T1486, T1083, T1055) costs 2 hours and makes every practitioner reviewer immediately favorably disposed.
